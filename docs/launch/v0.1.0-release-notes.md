# Setec v0.1.0

First public release. Setec is a Kubernetes-native operator that runs workloads inside Firecracker microVMs via Kata Containers. It is cloud-agnostic, self-hostable, and covered by Apache 2.0.

## Highlights

- **`Sandbox` custom resource.** Declare a microVM the same way you declare a `Deployment`. Setec materialises it as a pod with the `kata-fc` `RuntimeClass`, and the result is a hardware-isolated workload scheduled by the same control loop you already use.
- **Pre-warm pool with Firecracker snapshots.** Keep a configurable pool of paused microVMs ready on each node. Cold-start for a consumed pool entry is observed sub-100ms P50 on a prepared bare-metal host.
- **Multi-tenancy and observability.** Tenant identity from namespace labels or mTLS; per-Sandbox `NetworkPolicy`; Prometheus metrics and OpenTelemetry traces out of the box.
- **gRPC frontend.** A `SandboxService` API for programmatic consumers, mTLS-authenticated, tenant-scoped.
- **Self-hostable and cloud-agnostic.** No cloud-vendor dependencies. The only hard requirement is a KVM-capable Linux node with Kata Containers installed.

## Prerequisites

- Kubernetes 1.28 or later.
- At least one worker node with `/dev/kvm` (bare metal or nested-virt-capable VM).
- Kata Containers installed via `kata-deploy`; the `kata-fc` `RuntimeClass` must exist cluster-side.
- `helm` 3.8 or later.
- Cluster-admin for the install; subsequent use requires only namespace-scoped permissions.

## Installation

```bash
helm install setec oci://ghcr.io/zeroroot-ai/charts/setec \
  --namespace setec-system \
  --create-namespace
```

For a first walk-through, follow [`docs/getting-started.md`](../getting-started.md). The terse version lives in [`docs/quickstart.md`](../quickstart.md).

## What's Included

Refer to [`CHANGELOG.md`](../../CHANGELOG.md) for the full, phase-grouped feature list. The short version:

- Phase 1 shipped the MVP: `Sandbox` CRD, controller, RBAC, Helm chart, six E2E scenarios.
- Phase 2 added multi-tenancy, `SandboxClass`, metrics + tracing, validating webhook, node-agent DaemonSet, gRPC frontend.
- Phase 3 added snapshots, pause/resume, the pre-warm pool, the node-agent gRPC surface, and the Firecracker snapshot storage backend.
- Phase 4 closed the launch gap: the `setec-pool-vm` launcher, the pool reconcile tick, community docs, launch assets, supply-chain hygiene workflows, three example consumer programs, and a Grafana dashboard + Prometheus alert set.

## Known Limitations

- **Kata + Firecracker is a hard prerequisite.** Setec does not install or manage Kata. Clusters without a KVM-capable node will accept `Sandbox` resources but nothing will boot.
- **v1alpha1 API.** Both the CRD group and the gRPC frontend are `v1alpha1` and may change before `v1`. Consumers should expect a migration path between v0.x releases.
- **The pre-warm pool is node-local.** Cross-node restore-after-claim is not supported yet; classes with a pool must have nodes that can host the resulting workload.
- **Image prefetching uses the containerd v2 Go client.** The node-agent dials `/run/containerd/containerd.sock` (override via `--containerd-socket`), pulls into the `k8s.io` namespace by default, and accepts Docker-format registry credentials via `--containerd-auth-file`. Prefetch failures are classified into typed sentinels and surfaced via the `setec_node_image_prefetch_errors_total{reason}` counter.
- **Bare-metal E2E scenarios run on the maintainer's runner**, not in GitHub Actions. Three Phase 3 scenarios (pool cold start, storage-fill protection, upgrade-from-Phase-2) are expected to pass on the bare-metal runner; see [`docs/launch/v0.1.0-smoke-test-result.md`](./v0.1.0-smoke-test-result.md) for the recorded results at tag time.
